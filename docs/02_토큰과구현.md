# 02. 토큰과 구현 (구체적 값과 구현)

> 본 문서는 ERP No-Code 빌더 프로젝트의 **참조 문서**이다. 상위 원칙·규칙·색인은 「프로젝트 지침」을 따른다.
> 본 문서는 기술 스택, 디자인 토큰 값, 산출물 파일, 래퍼 패턴, 1차 원자 집합, 검증 방법을 다룬다.
> 값(색 hex·여백 px 등)은 잠정값이며 화면 검증 후 조정한다. 구조와 규칙은 「01. 규칙과 구조」를 따른다.

## 기술 스택과 각 도구의 역할

브라우저가 최종적으로 이해하는 것은 **HTML(뼈대) + CSS(스타일) + JavaScript(동작)** 셋뿐이다. 아래는 전부 "이 셋을 더 잘 쓰기 위한 중간 도구"다.

- **React** — 화면을 조각(컴포넌트)으로 짓는 방식. "상태가 이러면 이렇게 생긴다"를 선언하면 React가 알아서 다시 그린다. → 본 시스템 부품 철학의 세계관 그 자체.
- **JSX** — JavaScript 안에 HTML처럼 생김새를 쓰는 문법. 빌드 때 평범한 JS로 번역됨.
- **TypeScript (strict)** — 실행 전에 "여기 들어갈 값의 종류"를 검사하는 안전장치. **본 시스템의 "임의 판단 구역 좁히기"를 코드 단에서 강제하는 핵심 도구.** 잘못된 prop은 컴파일 단계에서 빨간 줄로 잡힌다. 사용자에겐 흔적도 안 남는다.
- **Mantine v8** — 이미 잘 만들어진 컴포넌트 + 토큰(theme) 시스템 묶음. **본 시스템의 베이스 층.** 단, 기본적으로 너무 열려 있어서(임의 hex/px 허용) 래퍼로 닫아야 한다.
- **Next.js (App Router)** — React 조각들을 주소별 페이지로 묶어 굴리는 틀. 폴더 구조 = URL. 인터랙티브 조각은 `'use client'` 표시.
- **빌드** — JSX/TS/흩어진 조각 → 브라우저용 HTML/CSS/JS로 번역·압축하는 공장.
- (Supabase PostgreSQL, Recharts, PWA — 지금 단계(시각 부품)와는 다른 층. 나중에 연결.)

> **베이스는 새로 고르는 게 아니라 이미 Mantine으로 정해져 있었다.**
> shadcn 등은 Mantine과 스타일 시스템이 충돌하므로 쓰지 않는다.
> 진짜 작업은 "Mantine을 어떻게 **닫느냐**"이다.

## 지금까지 내린 구체적 결정

### 디자인 토큰 (확정 — 단, 값은 화면에서 조정 가능)

- **primary** = 미지수 그릇. 현재 navy(hue 216). 10칸 사다리(50~900), index 6이 메인.
  - 회사가 바뀌면 이 사다리만 통째로 교체 → 전체 화면이 따라온다.
- **neutral** = `f(primary)`. primary와 같은 hue, 채도만 ~7%로 죽인 차가운 회색. 10칸(50~900).
  - primary가 바뀌면 neutral의 색감도 자동으로 따라온다. (관계만 박고 값은 안 박는다)
- **상태색(불변)** = success/warning/danger/info. 각 10칸. Tailwind의 대비 검증된 사다리를 출발점으로 사용.
  - warning은 갈색기 amber 대신 더 노란 yellow 계열.
- **간격** = 4px 베이스. xxs(4)/xs(8)/sm(12)/md(16)/lg(24)/xl(32)/xxl(48). 임의 px 금지.
- **타이포(6단계, 묶음)** = display/heading/subheading/body/**body-strong**/caption. 각각 {크기·굵기·행간} 묶음.
  - Mantine fontSizes는 크기만 담으므로, 6단계 묶음 전체를 `theme.other.typography`에 둔다(단일 진실 공급원).
  - 폰트: **PretendardGOV**(가변). `fontFamily="Pretendard GOV Variable"`(실제 패밀리명 — 공백 포함, 과거 `PretendardGOV` 붙임표기는 이름 불일치로 적용 안 됐던 오기). 단일 가변 woff2(전 weight 45~920)를 **패키지에 동봉(self-host)** 하고 `Providers`가 `fonts.css`로 로드 → CDN 없이 사내망·오프라인에서도 동작. 폴백은 `-apple-system`·시스템.
  - 이름은 **위계/스펙 축**으로 통일한다(용도 단어 금지). `body-strong` = body와 같은 크기 + 굵게(강조). 폼 라벨이 입력값(body)보다 작으면 위계가 거꾸로이므로 같은 크기 + 굵기로 구별한다.
  - **"라벨"은 타이포 단계 이름이 아니라 *역할* 이름이다.** 어느 단계를 쓸지는 컴포넌트(맥락)가 정한다 — 폼 라벨(FormField·Label 원자)은 `body-strong`(입력값과 동급 위계), 정보 라벨(DescriptionList)은 `caption`(값보다 보조). 색 역할(`text.primary`)이 맥락 따라 다른 값에 매핑되는 것과 같은 구조. (과거 `label`이 용도 이름이라 단계=역할 혼동을 일으켜 `body-strong`으로 개명.)
- **radius** = sm(8)/md(16)/full(9999). full은 Mantine 기본에 없어 커스텀 추가. (v0.10.0: squircle 곡률이 보이도록 sm 4→8·md 8→16 상향 — 작은 반경은 곡률이 안 드러남. 단계 수·키는 불변.)
- **모서리 곡률(squircle)** = `corner-shape: superellipse(2)`(애플식). radius 위에 *연속 곡률*만 얹는 단일 토큰(`--corner-shape`, theme.ts). **박스 컴포넌트에만 opt-in**(squircle.css 화이트리스트) — 원/알약(Avatar·Spinner·Switch·Chip·Radio 등)은 round 유지(squircle이 원을 둥근 사각형으로 각지게 만들기 때문). border·box-shadow·outline이 모양을 네이티브로 따라가 clip-path식 충돌 없음. 미지원 브라우저(Safari/FF)는 무시 → 평범한 둥근 모서리로 graceful fallback(radius 값은 전 브라우저 동일).
- **그림자** = sm(카드)/md(모달). none은 "안 줌".
- **보더** = 굵기 1px 한 종류 + 색은 neutral 참조(기본 200, 강조 300, 포커스 primary-600). 독립 토큰 아님.
- **아이콘 정렬 보정** = `--icon-baseline-shift`(기본 `-0.125em`). 아이콘 SVG의 광학 중심이 텍스트 baseline과 어긋나는 것을 보정하는 em 상대 상수(1/8 룰). **임의 px이 아니라 폰트 크기에 비례하는 토큰.** Icon 원자가 내장하여 텍스트와 함께 놓인 모든 아이콘이 자동 광학 정렬된다. 아이콘은 **SVG로 그린다(유니코드 글리프 금지)** — 유니코드는 브라우저·폰트마다 렌더가 달라 변동이 생기고, SVG는 viewBox를 통제하므로 환경 무관하게 일관(단 SVG도 정렬은 공짜가 아니라 위 보정 토큰이 함께 필요).
- **시맨틱 역할(텍스트·배경)** = 위 색 사다리를 *용도 이름*으로 한 겹 더 감싼 것. 컴포넌트는 사다리 칸(neutral-900)이 아니라 이 역할 이름을 참조한다.
  - 텍스트: `text.primary`(기본 본문·제목색) / `text.secondary`(보조·흐림) / `text.danger`(에러) / 필요시 `text.disabled` 등.
  - 배경: `bg.primary`(기본 면) / `bg.secondary`(표면) / `bg.tertiary`(페이지) 등.
  - **모드 분기는 이 역할 층에서 일어난다.** `text.primary`는 라이트=neutral 최상단(거의 검정), 다크=neutral 최하단(거의 흰색)으로 토큰이 매핑한다. 컴포넌트(Title 등)는 `text.primary`만 참조하고 검정/흰색 분기를 *알지 못한다*. (색이 뒤집히는 책임은 토큰이 지고, 컴포넌트는 역할 이름만 안다 — 에러 빨강을 입력칸이 boolean만 알고 실제 hex는 theme가 답하는 것과 같은 구조.)
  - 값(어느 칸에 매이나)은 화면에서 조정할 잠정값. 구조(역할 존재 + 모드 분기는 여기서)만 확정.

### 역할 변수 통로 (✅ 구현 확정 — 이번 세션)

역할에 따라 값이 바뀌는 모든 것(색·타이포·테두리·아이콘 정렬)을 단일 메커니즘 = **CSS 역할 변수**로 통일한다. 컴포넌트는 `var(--…)` 역할 이름만 부르고 실제 값·모드 분기를 모른다. (원칙: 「01」 강제 3층/역할 이름 참조. 구현 메커니즘은 본 절.)
- **모드 분기:** `cssVariablesResolver`(theme.ts)가 라이트/다크별로 `--text-*`/`--bg-*`/`--border-*`를 화면 최상단에 *깐다*. 원자는 읽기만 한다.
- **타이포:** 같은 resolver가 `--typo-{단계}-{size|weight|lh}`(6단계 묶음)를 방출. Text/Title/Label은 `var(--typo-body-size)` 식만 부른다(크기·굵기·행간을 모름).
- **상태색(에러):** FormField가 에러일 때 자식 영역에 `--field-border`를 danger 토큰으로 *덮는다*. 입력칸은 `var(--field-border, var(--border-default))`만 그려 자기가 에러인지 모른다.
- **아이콘 정렬:** `--icon-baseline-shift`(-0.125em)도 같은 통로 — Icon이 `verticalAlign`으로 소비.
- **모서리 곡률(v0.10.0):** `--corner-shape`(superellipse(2))도 같은 통로 — `squircle.css`가 박스 컴포넌트에서 `corner-shape: var(--corner-shape)`로 소비(원형 제외, 미지원 브라우저는 graceful fallback).
- 이 통로 외에 색·타이포를 부품에 주입하는 다른 길은 없다(임의 hex·style은 린트·타입이 차단).

### 산출물 파일 (✅ 검증 완료)

- `theme.ts` — 위 토큰을 Mantine `createTheme`로 담은 파일 + `cssVariablesResolver`(역할 변수 방출 — 위 「역할 변수 통로」; `--corner-shape` 포함). radius(sm8/md16/full)·`cornerShape`(squircle) 토큰의 단일 출처. `src/ui/` 안에 둔다.
- `mantine.d.ts` — 커스텀 색 이름, 추가 size 키(full/xxs/xxl), theme.other 타입을 TypeScript에 선언. **자동완성 힌트(지도) 역할.** tsconfig include에 잡혀야 함.
- `eslint.config.mjs` — 강제 규칙(그물 2). 문 1(Mantine 직접 import 금지) + 문 2(hex 금지).
- **격리 구조** — Mantine에 닿는 모든 것(컴포넌트·CSS·Provider·ColorSchemeScript)은 `src/ui/` 안에. 바깥은 배럴(`src/ui/index.ts` → `@/ui`)만 import. Provider 배선도 `src/ui/Providers.tsx`에 둔다.
- `Providers.tsx` — MantineProvider에 theme + `cssVariablesResolver` + `defaultColorScheme` 주입. CSS import도 여기(격리): `fonts.css`(폰트) · `@mantine/core/styles.css` · `@mantine/dates/styles.css` · `@mantine/notifications/styles.css` · `squircle.css`(모서리 곡률). 토스트 `<Notifications position="top-right"/>`도 마운트.
- `squircle.css` (v0.10.0 신설) — 박스 컴포넌트에 `corner-shape`(애플식 squircle)를 opt-in으로 적용하는 전역 CSS. 화이트리스트(Button·Paper·Input·Modal·Popover·SegmentedControl)라 원형(Avatar·Spinner·Switch·Chip·Radio 등)은 자동 제외. Providers가 로드(격리, 헌법 7).
- `src/ui/fonts.css` + `src/ui/fonts/PretendardGOVVariable.woff2` — PretendardGOV 가변 폰트 self-host(@font-face). 패키지 동봉이라 CDN 의존 0(사내망 OK). 5.4MB(전 weight 단일 파일) — 용량을 내주고 차단 무관 동작을 산 트레이드오프(내부망 맥락에선 LAN+swap+캐시로 체감 작음).
- `_fieldStyles.ts` — 입력칸 테두리 통로(`var(--field-border, var(--border-default))`)를 입력 원자들이 공유.
- `_cells.tsx` — 유기체 공유 패턴(셀 type enum·`Action` 타입·`renderAction` 헬퍼). 부품 격리 구역 안.
- `src/schema/` — **데이터 세계**(부품 격리 구역 `src/ui`와 분리). `fields.ts`(FieldSpec·FieldType — pattern/mask/조건부 포함) + `validation.ts`(buildZodSchema·isFilled, Zod) + `index.ts`. 도메인 진입로.

- `src/ui/notify.tsx` — 토스트 명령형 API. `@mantine/notifications`를 격리(바깥은 notify만). Providers에 `<Notifications/>` 마운트.
- 신규 의존성: `@mantine/notifications`(토스트용 공식 패키지, src/ui 격리).
- `src/app/customers/` — **첫 수직 슬라이스**(데모 앱, 라이브러리 아님). 목록(`page.tsx`)·상세(`[id]/page.tsx`)·데모 도메인(`_data.ts` — FieldSpec[]·rows·mockResolvers(patch)). 부품은 `@/ui`·`@/schema`만 import.

**검증된 환경 디테일 (재현용):**

- Mantine v8은 `postcss.config.cjs` 필요. hydration 경고는 `mantineHtmlProps`로 해결(본 DSL과 무관한 Mantine SSR 배선).

### 패키지·배포 (✅ GitHub Packages publish 완료 — `@byeongjunkim-jjaim/erp-dsl`)

라이브러리를 **GitHub Packages npm 레지스트리에 publish하는 진짜 패키지**로 만들었다. 소비 레포(ERP)는 이걸 **import만** 하고 절대 수정하지 않는다 — 카탈로그 확장은 본 라이브러리 레포에서만(헌법 4가 레지스트리 경계로 강제됨).

- **이름**: 스코프형 `@byeongjunkim-jjaim/erp-dsl`(GitHub Packages는 스코프=오너명 필수, 소문자). `publishConfig.registry=https://npm.pkg.github.com`. 현재 버전 **0.10.0**.
- **소스 배포 + transpilePackages**: 빌드 산출물(dist) 없이 `.tsx` 원본을 그대로 publish하고, 소비 앱(Next)이 `transpilePackages: ['@byeongjunkim-jjaim/erp-dsl']`로 컴파일. `exports`: `.`(부품 `src/ui/index.ts`) / `./schema`(`src/schema/index.ts`).
- **peerDependencies**: `@mantine/*`·`react`·`react-dom`·`dayjs`·`zod`는 소비 앱이 설치(패키지엔 dependencies 0). 로컬 빌드용으론 devDependencies에도 둠.
- **publish 대상 격리**: `files` 화이트리스트(`["src/ui","src/schema","README.md","!src/ui/_dev.ts","!src/ui/_Dev*"]`)로 publish 셋 = `src/ui`(부품) + `src/schema` + 폰트 woff2 + README. 데모 앱(`src/app`)·문서(`docs/`)는 화이트리스트 밖이라 자동 제외, dev 갤러리(`_dev.ts`·`_Dev*`)는 명시 제외. dev 갤러리는 공개 배럴(`index.ts`)에서도 빼고 `_dev.ts`/`_registry.tsx`로 분리(데모만 사용). ### 래퍼 패턴 (✅ 구현·검증 완료 — Button)

각 Mantine 컴포넌트를 얇게 감싸 열린 prop을 잘라낸다. **핵심: Mantine 타입을 상속하지 않고 Props를 손으로 쓴다** (그래야 `(string & {})` 탈출구가 안 딸려온다).
```tsx
// src/ui/Button.tsx (요지)
type ButtonVariant = 'primary' | 'secondary' | 'danger' | 'ghost';
type ButtonSize = 'sm' | 'md';
type ButtonProps = {              // Mantine ButtonProps를 extends 하지 않음 (의도)
  variant?: ButtonVariant; size?: ButtonSize; children: ReactNode;
  leftIcon?: ReactNode; rightIcon?: ReactNode; loading?: boolean;
  disabled?: boolean; fullWidth?: boolean; type?: 'button' | 'submit';
  onClick?: () => void;
  // className / style 은 의도적으로 노출 안 함 (우회 차단)
};
const VARIANT: Record<ButtonVariant, { color: string; mantineVariant: string }> = {
  primary:   { color: 'primary', mantineVariant: 'filled' },
  secondary: { color: 'neutral', mantineVariant: 'default' },
  danger:    { color: 'danger',  mantineVariant: 'filled' },
  ghost:     { color: 'neutral', mantineVariant: 'subtle' },
};
// radius="sm" 등은 래퍼 내부에 정책으로 고정. 바깥에서 못 바꾼다.
```
- 래퍼는 색 *역할 이름*(`'primary'`)만 안다. navy 같은 실제 hex는 모른다(theme가 답한다).
- 바깥에선 `<Button variant="primary">`만 쓸 수 있다. color/px/radius/className은 못 넘긴다.
- 닫힘은 "바깥에 노출하는 경계(Props)"에서 일어난다. 래퍼 *내부*는 Mantine 열린 API를 만져도 된다(예: `mantineVariant: string`은 느슨해도 경계가 아니라 내부라 OK).
- 나머지 원자(Input, Badge, Select…)는 전부 이 틀의 복붙이다.

### 1차 원자 집합 (확정 — 분류·기준 완료.)

**의미 원자** (도메인 무관, 범용. "의미 고유성"으로 등록 — 빈도 아님):
- 입력: `TextInput` `NumberInput` `CurrencyInput` `Textarea` `PasswordInput` `Select` `Checkbox` `Radio` `Switch` `DatePicker` `MultiDatePicker`
- 표시·행동: `Button` `Badge` `Chip` `Text` `Title` `Label` `Anchor` `Icon` `Avatar` `Tooltip` `Popover` `Spinner` `SegmentedControl` `TabBar` `Image`

**레이아웃 원자:** `Card`(variant로 elevated/outlined/flat 닫음) `Divider` `Container`(maxWidth로 콘텐츠 폭 천장 결정)
**배치 프리미티브:** `Stack` `Group` `Grid` (prop·기본값은 4-1)

확정된 판별 디테일:
- Input이 셋(Text/Number/Password)인 이유: *생김새가 아니라 받는 데이터·행동이 다름*. variant가 아니라 별개 원자.
- `Switch`(on/off 한 개) ≠ `Tabs`(화면 전환, 유기체).
- `Text`(본문 3: body/body-strong/caption) / `Title`(제목 3: display/heading/subheading) — 6단계를 나눠 가짐.
- `Label` = "무언가의 이름표"(입력칸 전용 아님). 항상 짝이 있어야 함(Text는 혼자 섬).
- `Badge`(표시 전용) ≠ `Chip`(행동 있음: 선택/삭제).
- `Anchor` = 이동 의미 고유 → Button만으론 대체 불가라 등록(빈도 낮아도).
- `MultiSelect`는 원자 아님 → Select+Chip *분자*. `로딩버튼`도 Button+Spinner 분자.
- `Spinner`는 독립 원자(Button의 loading과 별개 — 페이지/영역 로딩용).
- `StepperBar`/`ProgressBar`는 원자 아님 → 노드+연결선의 *유기체*(연결선은 원자가 아니라 유기체 내부 배치).
- `Popover` = Tooltip(hover·텍스트)의 형제. trigger가 클릭이고 content가 *슬롯(부품)*인 점이 다름. content는 raw ReactNode 슬롯으로 **Modal children과 동형**(방식 A — Popover는 안이 뭔지 모름). 자식을 받지만 의미 원자 옆에 두되 "content 슬롯을 받는 예외 원자"로 명시. 안의 스타일 우회는 인지(부품만 조립)로 막고 hex는 eslint가 차단.
- `Image` = 콘텐츠 이미지(제품사진·도면썸네일·로고). Avatar(사람 식별용 원형·이니셜 폴백)와 별개 — 임의 콘텐츠 이미지의 고정 종횡비 박스. 자식 안 받음(순수 의미 원자).

원자 판별 기준·경계 닫기 규칙·가변/고정은 「01. 규칙과 구조」 참조. 각 원자의 개별 prop은 아래 「원자 prop 규격」 절에서 확정(Button 포함 **25개** 의미 원자 + 레이아웃 3 + 프리미티브 3 — prop 표 실측 기준. Stage5에서 Popover·Image 추가 22→24, v0.3.0~에서 MultiDatePicker 추가 24→25; 0.11~0.16에서 Skeleton·Progress·Combobox·TimePicker·CountBadge·CurrencyInput 추가 → **31**(신규 6종 prop 규격은 `_catalog`·README 5절·「04」가 단일 출처). 원자 전부 래퍼 구현·검증 완료(tsc·eslint·강제 발화).

## 원자 prop 규격 (✅ 확정 — 구현·검증 완료)

> 「1차 원자 집합」에서 분류·판별이 끝난 각 원자의 노출 prop을 확정한다. 컬럼은 `prop / 범주(스타일·콘텐츠·기능) / 값 / 노출`. ❌(미노출)에는 *왜 닫는가*를 적는다 — 래퍼 고정 / FormField 소유 / 스키마로 / 규칙 3 / 경쟁 경로 제거 등.
>
> 범주는 「01. 규칙과 구조」 4-B의 선결 분류(스타일·구조 / 콘텐츠 / 기능·데이터)를 따른다. 닫기 규칙은 스타일/구조 prop에 걸리고, 데이터 prop엔 규칙 2(경쟁 경로)만 걸린다.

### 입력군을 가로지르는 횡단 규칙 4개

입력군 9개는 아래 4개 규칙을 공통으로 적용한 뒤, 원자별 고유분만 더한다. 각 표의 ❌ 사유는 이 규칙들의 약칭이다.

1. **장식·검증의 표현은 FormField(분자)가 소유** — `label`·`description`·`withAsterisk`·별표는 물론, **에러 표현(메시지 + 에러 시 빨간 테두리)도** 입력 원자에서 벗긴다. 입력칸은 입력에만 집중한다. (입력칸은 자기가 에러인지 모른다 — 테두리 색은 FormField가 CSS 역할 변수로 덮어 그린다. → 「FormField」 절)
2. **controlled 전용** — `value`/`onChange`(또는 `checked`)만 노출하고 `defaultValue`/`defaultChecked`(uncontrolled 입구)는 벗긴다. 상태 주인은 렌더러 하나여야 한다.
3. **검증의 진실은 스키마(Zod)로** — "이 필드가 받을 값의 제약"(NumberInput `min`/`max`/`step`, Select `clearable`=비울 수 있나, DatePicker 선택 가능 날짜 범위)은 원자 prop에 두지 않는다. 원자에도 두면 같은 진실이 두 곳(원자+스키마)에 생긴다. **error도 검증의 산물이므로 같은 이유로 원자 밖이다** — min/max를 스키마로 민 것과 동일한 처리.
4. **`className`·`style` 안 노출(규칙 3) + raw `ReactNode` 구멍 안 엶** — 어댄먼트(`leftSection`/`rightSection`)·인라인 라벨 등은 닫거나 `string`으로 좁힌다. 정말 필요해지면 raw 노드가 아니라 `unit?: string`처럼 좁힌 형태로 연다(rule of three).

> **`size='full'`(v0.11.0):** 고정 px이 아니라 *부모가 분배한 폭*(100%)을 받되 4:3 비율만 잠근다 — 임의 px이 아니라 닫힌 값 하나. ObjectCard 카드(상세) 뷰의 풀폭 배너 썸네일용. "폭은 위에서 분배" 원칙과 오히려 정합(고정치수 예외에서 한 발 더 헌법 쪽으로).

### FormField (✅ 확정 — 구현·검증 완료)

입력 원자에서 벗긴 장식·에러 표현의 수령자. 입력 컨트롤을 `children`으로 받아(방식 A — FormField는 가운데가 뭔지 모른다) 위아래에 장식을 두른다.

조립 구조 (Stack, gap=토큰): `label`+별표 → `children`(컨트롤) → `error` 메시지(있을 때만). (`description`은 두지 않음 — 아래 참조.)

정렬·간격 default (전부 프리미티브+토큰, 손 px 0):

모든 자식(label·input·error)은 같은 왼쪽 기준선에서 시작(Stack align=stretch). 입력칸 placeholder가 안쪽으로 보이는 것은 입력칸 자신의 밀도(size) 패딩일 뿐 정렬이 아니다.
중첩 Stack으로 구간별 gap을 다르게 준다(균일 gap 아님): 바깥 Stack(gap=xs)에 [필드 묶음, error]를 담고, 안쪽 Stack(gap=xxs)에 [label, input]을 담는다. 라벨·입력칸은 한 덩어리로 붙고(xxs), 에러는 그 아래로 떨어진다(xs).
description prop은 두지 않는다 — 설명은 입력칸 placeholder로 경로를 통일한다(desc와 placeholder는 경쟁 경로). 지속 도움말이 필요한 필드는 별개 분자의 후보이며 rule of three 충족 시 만든다.

| prop | 범주 | 값 | 노출 |
|---|---|---|---|
| `label` | 콘텐츠 | `string` | ✅ |
| `withAsterisk` | 스타일 | `boolean`(별표 *표시*만 — 필수 *검증*은 스키마) | ✅ |
| `error` | 콘텐츠 | `string`(메시지. 있으면 에러 상태로 판정) | ✅ |
| `children` | — | 입력 컨트롤(원자) | ✅ |
| `className` `style` | — | 규칙 3 | ❌ |

고정(pin)하는 상태 — 분자가 붙박이로 만들어 통일성을 강제하는 것:
- label = 타이포 `body-strong` 단계 + `text.primary` 고정
- 별표 = `text.danger` 고정
- error 메시지 = `caption` + `text.danger` 고정
- 조각 간 gap = 토큰 고정

에러 분배 (핵심 메커니즘):
- 사용자는 `error` *메시지 하나만* 준다. FormField가 "메시지 있음 = 에러 상태"로 **1회 판정**한다.
- 메시지는 FormField가 직접 그린다(맨 아래, danger).
- 빨간 테두리는 입력칸이 그리되, **입력칸은 자기가 에러인지 모른다.** 입력칸 테두리는 `--field-border`라는 CSS 역할 변수로 그려지고(없으면 neutral 토큰 fallback), FormField가 에러 상태일 때 자식 영역에 `--field-border: [danger 토큰]`을 깐다.
- prop 구멍이 아니라 CSS 역할 변수다 — 바깥에서 임의 색을 못 넣고(FormField가 neutral/danger 토큰만 넣음), 입력칸 Props엔 테두리 관련 prop이 없다(닫힘 유지). "컴포넌트는 역할만, 값은 문맥·토큰이"의 색 토큰과 동일 구조.

Select + Chip 결합. value 배열을 받아 내부에서 Select+Chip을 조립(내부 조립형 — children 아님). 고른 값이 Chip으로 쌓인다.

고정하는 상태:
- 고른 값을 **Chip(color=neutral 고정)**으로 표시. Chip 원자는 가변이나 MultiSelect가 neutral로 고정해 가린다(통일).

- **항목 선택 시 드롭다운 유지(닫지 않음)** — 단일 Select는 고르면 닫히지만, 다중은 연속 선택이 본질이라 분자가 "열린 채 유지"를 동작으로 고정한다. (분자가 색·모양뿐 아니라 *동작*도 고정하는 사례.)
- 드롭다운 체크 표시는 Icon 원자(SVG + baseline 보정), Group(align=center).

Button을 **아이콘 전용으로 고정**한 named 분자(「01」 4-C가 예고한 "아이콘형"의 실현 — 같은 Button 원자에 자식을 텍스트 대신 아이콘으로). 텍스트가 없으므로 의미를 `label`(aria-label)로 보존한다. 정사각. 행 액션의 "윤곽 없는 빨강 삭제", 프로필 더보기(dots) 등에 쓴다.

| prop | 범주 | 값 | 노출 |
|---|---|---|---|
| `icon` | 콘텐츠 | `IconName`(Icon 세트 식별자) | ✅ |
| `label` | 콘텐츠 | `string`(aria-label 필수 — 아이콘만이라 의미를 텍스트로 보존) | ✅ |
| `variant` | 스타일 | `'primary' \| 'secondary' \| 'danger' \| 'ghost'` | ✅ |
| `size` | 스타일 | `'sm' \| 'md'` | ✅ |
| `disabled` | 스타일 | `boolean` | ✅ |
| `onClick` | 기능 | 콜백 | ✅ |
| 외형(정사각·radius·색 매핑) | — | 고정(variant→외형·Icon 색 역할. danger=윤곽 없는 빨강 subtle) | ❌ |
| `className` `style` | — | 규칙 3 | ❌ |

고정: variant가 (외형, Icon 색 역할)을 함께 고정 — primary=filled, secondary=default, danger=subtle+danger색, ghost=subtle+secondary색. `renderAction`이 `iconOnly` 액션을 이 분자로 라우팅한다.

### Callout (✅ 확정 — 구현·검증 완료)

인라인 문맥 메시지 — 그 자리에 머무는 **비휘발** 안내(중복 연락처 경고·레거시 주소 힌트 등). 토스트(휘발·우상단)와 의미가 다르다: Callout은 폼/구획 안에 박혀 사라지지 않는다. Group + Stack + Icon + Text 조립.

|---|---|---|---|

| `children` | 콘텐츠 | `string`(본문 — raw 노드 금지) | ✅ |
| `icon` | 콘텐츠 | `IconName`(선택; 생략 시 tone별 기본 아이콘 내부 고정) | ✅ |
| `className` `style` | — | 규칙 3 | ❌ |

고정: 배경=`var(--mantine-color-{tone}-0)`·보더=`-2`·아이콘=`-7`(의미색 스케일 토큰, raw hex 0). 패딩 md·radius md. tone→기본아이콘 내부 매핑(info→info, warning→alert-triangle, danger→alert-circle, neutral→info). **텍스트는 primary/secondary 역할 통로 유지**(연한 tone 배경 위 틴트 텍스트는 대비 저하 — tone은 배경·보더·아이콘이 전달). 아이콘 광학 보정은 flex라 `vertical-align` 대신 `transform`으로.

> **경계(중요): 헤더+본문 한 단위까지만.** "여러 개 중 하나만 열림(accordion)"은 이 부품 일이 아니다 — 여러 개를 쌓아 구성(§11-3 "옵션 쌓지 말고"). header 슬롯 안 인터랙티브 요소(StatusRow 액션 등)는 호출측이 전파 차단(ObjectCard actions 동형). 실수요: 정산 발주별 요약행→품목+소계, 발주 완료섹션 헤더→완료 카드들(기본 접힘).

## 유기체 prop 규격 (✅ 유기체 13 — 아래 9종 prop 표 + 0.11 추가 4종[Drawer·Stepper·Transfer·ToastHost]은 `_catalog`·README 5절·「04」가 단일 출처)

> 유기체 = 화면 한 구획 차지, 도메인 스키마를 입력받아 작동(「01」). 헌법 1번이 가장 빡세게 적용 — 도메인은 컴포넌트로 안 들어오고 스키마(데이터)로만. 측량(kk) 빈도가 우선순위를 안내했을 뿐, 정의는 도메인 무관. 측량의 도메인 박힌 컴포넌트(OrderView 등)는 부품 아닌 *스키마*(DataTable+발주 스키마).

### 공통 패턴 — 셀/값 표현 타입 enum (현재 **14종**)
DataTable·DescriptionList가 "값을 어떤 표현으로 그리나"를 공유(`_cells.tsx`의 `CellType`). 자유 render 함수 금지(raw 구멍). 새 타입은 사람이 큐레이션으로만 추가(헌법 4번). v0.2.0 7종에서 ERP 실수요로 7종 더 추가 → 현재 14종:
- 기본 7: `text` / `badge`(값→색 매핑 스키마 주입) / `number`(천 단위 콤마) / `currency`(₩ + 콤마, KR locale 잠정) / `date` / `boolean`(true→Icon `check`(primary), false→엷은 대시 `—`) / `actions`(`Action[]`, 셀에선 sm).
- 추가 8: `user`(`{name, src?}` → Avatar sm + 이름) / `tags`(`string[]` → neutral Badge 묶음) / `link`(`{label, href}` → Anchor) / `percent`(`number` → `N%`) / `secondary`(`{primary, secondary?}` → 본문+보조 2줄) / `relative-time`(ISO → "3일 전" 상대시간, dayjs ko) / `thumbnail`(src → Image sm) / `chevron`(`boolean?` → 디스클로저 `›` — "행=상세로 인도" 시각 장치, 자체 동작 없음·클릭은 행 onRowClick으로 버블).
- 정렬: number/currency/percent/actions는 우측정렬, 나머지 좌측/기본. 통화·숫자 포맷은 `fmtCurrency`/`fmtNumber` 단일 출처(SummaryCard·TotalRow도 재사용).

### 공통 패턴 — actions 배열 + `renderAction` 헬퍼
Modal·PageHeader·EmptyState·DataTable·DetailPage가 버튼을 `Action` 형태로 받고 *배치는 컴포넌트가 고정*한다.

`Action` = `{ label: string; variant?: 'primary'|'secondary'|'danger'|'ghost'; onClick: () => void; icon?: IconName; iconOnly?: boolean }`
- `icon` — 좌측 아이콘(행위 중심 액션). 중립 액션(취소·이동)은 생략 → 텍스트만.
- `iconOnly` — true면 아이콘 전용 버튼(IconButton)으로 렌더. `icon` 필수, `label`은 aria-label로 보존(접근성). 행 액션의 "윤곽 없는 삭제" 등.

**`renderAction(action, key, size='md')`** — 모든 호출처 공유 단일 헬퍼(`_cells.tsx`). 분기: `iconOnly+icon` → IconButton / `icon+label` → Button(leftIcon) / 그 외 → 텍스트 Button. **PageHeader·DataTable 셀은 `size='sm'`로 호출**(헤더는 제목이 주인공·표는 밀도). 닫힌 enum만 받으므로 자유 노드 구멍 없음. (ActionGroup 분자 후보 — rule of three 충족이나 배치 맥락이 미묘하게 달라 구현 시 추출 판단. 「03」)

### Modal (✅ 확정)

도메인 무관 껍데기. 본문(children)에 도메인 폼이 오되 Modal은 모름(방식 A).

| prop | 범주 | 값 | 노출 |
|---|---|---|---|
| `opened` | 기능 | `boolean`(controlled) | ✅ |

| `title` | 콘텐츠 | `string` | ✅ |

| `size` | 스타일 | `'sm' \| 'md' \| 'lg' \| 'xl' \| 'full'`(폭 — sm/md/lg/xl=Mantine 토큰, **full=95vw·90vh**, 풀스크린 아님·radius/여백 유지) | ✅ |
| `closeOnOverlayClick` | 스타일 | `boolean`(폼 모달서 끄게) | ✅ |
| `children` | — | 본문(도메인, Modal은 모름) | ✅ |
| 스크롤 구조(헤더·푸터 고정·본문 스크롤) | — | 고정. 3영역(헤더/본문/푸터) flex 세로. 헤더·푸터 `flex:none`(고정), 본문 `flex:1 + overflowY:auto`(긴 내용일 때만 스크롤), 컨테이너 `maxHeight:85vh`(full=90vh)로 캡. Mantine 본문 패딩(`padding=0`) 끄고 패딩 주인을 우리 3영역으로 이관 | ❌ |
| 푸터 배치 | — | 고정(오른쪽 정렬, primary/danger 맨 오른쪽, gap=xs, 배열 순서 무관) | ❌ |
| 헤더(제목+X) | — | 고정. Mantine 기본 헤더(`title` prop) 미사용 → 우리 `Group(justify=between, align=center)` + Title + Icon `x`로 직접 조립 | ❌ |
| 오버레이·중앙배치·그림자(md) | — | 고정. 그림자 토큰 `md`(02 "모달용"). 카드(sm)보다 강해 떠 보임 | ❌ |

> **헤더 정렬은 우리 프리미티브 보장.** 제목·X를 Mantine 내부 헤더(`title`)에 위임하면 정렬이 Mantine CSS 지배라 우리 보장이 끊긴다(미세 세로 어긋남 발생). `withCloseButton={false}`로 기본 헤더를 끄고 우리 Group으로 직접 조립한다. (서드파티 내부 레이아웃 불신 — 「01」 따름정리.)

> **스크롤은 직접 조립(명시 예외 2건).** ① 본문만 스크롤하는 3영역 flex는 Stack 프리미티브가 노출하지 않는 표현(`flex:none`/`flex:1`/`overflowY`/`maxHeight`)이라 src/ui 격리 구역 내 Modal 한 파일의 **raw flex**가 된다 — 이는 "서드파티 내부 레이아웃 불신" 따름정리(헤더 직접 조립)의 *재적용*이며, 사례가 Modal 하나뿐이라 Stack 확장(프리미티브 손대기)은 rule of three 미달로 보류. ② `maxHeight: 85vh`는 공간 장부에 처음 들어오는 **잠정 단위(vh)** — 뷰포트 캡은 vh가 자연 단위라 수용, 값은 화면 검증에서 조정. xl size는 보류(요청).

푸터: actions를 variant 보고 배치 고정(primary/danger 끝, secondary 왼쪽). 주 행동 모달당 하나 전제. 닫기 X는 Icon 원자. 유형 자동(폼/확인/정보)은 패턴 굳으면 별도 분자(ConfirmModal).

## 피드백 시스템 — Toast (`notify`) (✅ 확정 — 구현·검증 완료)

휘발 피드백 — 우상단에 잠깐 떴다 사라지는 작업 결과(저장됨·삭제 실패 등). **부품이 아니라 명령형 트리거**라 페이지 스키마에 안 들어간다(LLM 스키마 표면 밖, resolver·onClick과 같은 *코드 배선* 자리). 그래서 "LLM은 스키마, 렌더러는 그림" 원칙을 안 깬다.

- API: `notify.success(p)` / `notify.danger(p)` / `notify.warning(p)` / `notify.info(p)`. `p`는 `string` 또는 `{ title?, message }`.
- 닫힘: tone = success/danger/warning/info(역할 enum) + 메시지 string. tone→(의미색 토큰·아이콘) 내부 매핑. **위치(우상단)·지속시간(4s)·애니메이션은 시스템 고정**(`Providers`의 `<Notifications/>`).
- 격리: `@mantine/notifications` import·Provider 배선은 `src/ui` 안에만. 바깥은 `notify.*`만 호출(헌법 7). **신규 의존성**: `@mantine/notifications`(공식 패키지).
- **경계(중요)**: 토스트는 **서버/작업 레벨 결과**(성공·작업 오류)만. **필드 검증 표시는 인라인 FormField 몫**(토스트로 안 띄움 — 「03」 검증 흐름). 둘이 경쟁하지 않는다.



> **부품은 어휘, 스키마는 문법**(「00」). 스키마 층은 부품이 아니다(원자/분자 분류에 안 들어감). LLM이 생성하고 Zod가 검증하는 "한 입력칸의 명세(데이터 shape)"다. **도메인은 오직 여기로만 들어온다(헌법 1번)** — `FieldSpec[]`이 곧 도메인의 진입로. 산출물 위치는 `src/schema/`(부품 격리 구역 `src/ui`와 분리 — 데이터 세계).

### FieldSpec (한 입력칸의 명세)

```ts
type FieldType = 'text' | 'number' | 'textarea' | 'select' | 'date' | 'checkbox' | 'lookup';
type FieldSpec = {
  name: string;                                  // 폼 값 키
  label: string;
  type: FieldType;
  required?: boolean;
  placeholder?: string;
  options?: { label: string; value: string }[]; // select 전용 (원자 options와 같은 형태)
  lookupKey?: string;                            // lookup 전용: 어떤 resolver를 쓸지 식별(문자열 — 함수 아님)
  span?: 1 | 2;                                   // FormSection columns=2일 때 한 줄 전체(기본 1)
  pattern?: string;                               // 정규식(데이터). buildZodSchema가 .refine(regex)로 검증. 잘못된 패턴은 가드.
  mask?: 'phone';                                 // 입력 마스킹(명명 enum, 임의 금지 — 큐레이션). FormSection이 onChange에서 적용.
  requiredWhen?: { field: string; filled: boolean }; // 조건부 필수(선언형, 함수 금지). .superRefine 교차검증.
  disabledWhen?: { field: string; filled: boolean }; // 조건부 비활성. FormSection이 토글.
};
```

- **함수(resolver 등 배선)는 스키마에 두지 않는다** — 직렬화 불가 + LLM이 생성 불가. lookup은 `lookupKey`(문자열)로만 가리키고, 실제 조회 함수는 코드 측(앱)에서 `resolvers`로 주입한다. 부품은 그게 "카카오 주소조회"인지 모른다 — "외부 조회"만 안다.
- **`lookup` 타입**: 외부 조회로 채워지는 입력(주소·우편번호·사업자조회 등). 입력칸이 행을 채우고 우측에 검색 버튼이 붙는다(FormSection이 조립).
- **`pattern`(정규식)은 우리 시스템의 유일한 열린 스칼라**다. 다만 *컴포넌트 styling prop*(헌법 5가 막는 대상)이 아니라 **스키마/데이터 층의 필드**라서, `label`·`placeholder`가 string인 것과 같은 성격(도메인이 합법적으로 들어오는 자리). `new RegExp(pattern)`이 잘못된 패턴에 throw하지 않도록 **가드**(`safeRegExp`).
- **`mask`와 `pattern`은 별개 축**: 마스킹(입력 중 하이픈)은 *보이는 형태* 변환, pattern은 *검증*. 같은 필드가 둘 다 가질 수 있다(전화처럼). 마스크 카탈로그(`_masks.ts`, src/ui)는 사람이 큐레이션으로만 늘린다(헌법 4) — LLM은 명명 enum에서 **고르기만**, 정규식 마스크를 쓰는 열린 구멍이 아니다.
- **`requiredWhen`/`disabledWhen`은 선언형**(함수 금지 — 직렬화·LLM 생성 가능). 술어는 현재 `filled` 하나(참조 필드가 채워졌는가). 예: `detail.requiredWhen={field:'address',filled:true}`, `detail.disabledWhen={field:'address',filled:false}`.
- FieldSpec은 직렬화 가능한 순수 데이터 → LLM이 생성하는 단위이자, Zod 검증의 입력이다.

### buildZodSchema (강제 3층의 스키마판)

`buildZodSchema(fields: FieldSpec[])` → Zod 객체 스키마를 도출한다. "LLM이 뱉은 데이터가 렌더러에 가기 전 런타임 관문". 타입별 매핑:
- number → `z.number()`, checkbox → `z.boolean()`, select → `z.enum(options.value들)`, text/textarea/date/lookup → `z.string()`.
- `required`면 그대로, 아니면 `.optional()`. required string은 `.min(1, "${label}은(는) 필수입니다")`.
- **`pattern`** → `.refine(v => v==='' || regex.test(v), …)`(빈 값은 통과 — 필수 여부는 별도). 잘못된 pattern은 무시(가드).
- **`requiredWhen`** → 마지막에 `.superRefine`으로 교차검증: 참조 필드의 filled 상태가 조건과 일치하고(=활성) `disabledWhen`으로 비활성이 아니면, 비어 있을 때 이슈 추가.
- `isFilled(v)` 단일 헬퍼(빈문자/null=false, boolean은 값 자체, 그 외 true)를 **조건부 술어와 검증이 공유**(단일 정의).
- **검증의 진실은 여기 하나**(원자에서 min/max·error를 벗긴 이유와 같은 단일 출처 — 「01」 입력군 횡단규칙 3). select enum이 옵션 값에서 자동 도출되어 원자·스키마 이중 정의가 없다.

> 타입 7종 + required + 정규식 + 조건부 필수까지 구현·검증(런타임 probe로 정규식 위반·조건부 발화·비활성 면제·패턴 가드 확인). min/max·교차필드(끝≥시작)·다중 술어는 데이터 계약 후 확장. 강제 3층(타입·린트)과 같은 사상의 *데이터 버전*이다.

고정(코드): ① 타입→원자 매핑(text→TextInput, number→NumberInput, textarea→Textarea, select→Select, date→DatePicker, checkbox→Checkbox, lookup→TextInput+검색버튼). ② 모든 필드를 FormField로 감싸(label·error 분배). ③ 배치=1열 Stack / 2열 Grid(`span`으로 한 줄 전체 허용), gap=토큰. ④ **마스크**: `field.mask` 있으면 onChange에서 `applyMask` 적용(text 계열). ⑤ **조건부**: 매 렌더 현재 값으로 `effectiveRequired = required || requiredWhen 충족`, `effectiveDisabled = disabledWhen 충족` 계산 → withAsterisk·disabled 토글. ⑥ **lookup resolver**: `apply(patch)`로 여러 필드를 한 번에 채움(단일 set 폐기 — 경쟁 경로 차단). 가변(스키마): 어떤 필드가 몇 개·무슨 타입·라벨·필수·options·span·pattern·mask·조건부. **errors는 부모가 Zod로 판정해 주입** — FormSection은 검증하지 않고 분배만(검증의 진실은 스키마, 「01」).

> **우측 영역 재설계(v0.11.0):**
> - **폴더(상위 분류) 선택 → 하위 분류를 클릭 가능한 폴더 타일**(빈 상태 텍스트 폐기 — 폴더는 하위를 드러내야 로직상 맞다). 타일 클릭=드릴다운. 잎(자식 없음)만 품목 보유(잎-전용 모델).
> - **다중 뷰**: 카드(상세)/목록(요약) `SegmentedControl` 토글(object 보일 때만 노출 — 폴더엔 숨김). 품목 카드 = `ObjectCard layout`. **고정 4열**(`repeat(4)`=span 3, 아이템 수로 동적 분할 안 함). 폴더 타일도 같은 4열.
> - **전역 검색**: 헤더 좌측 도구. 쿼리 시 결과 모드(각 결과에 경로 Breadcrumb, 클릭=집 점프+검색 해제). 필터는 소비처(`searchResults` 주입). 중복은 불변식상 0.
> - **자동 펼침**: 우측에서 노드로 점프(폴더 타일·브레드크럼·검색결과) 시 좌측 트리의 접힌 조상을 자동으로 펼친 뒤 선택(`onToggle`은 접힌 것만 가드).
> - **헤더 안정성(→「03」 헤더 안정성 원칙):** **3존(v0.14.0)** — `[breadcrumb 제목존: flex:1 말줄임] [도구: 고정] [품목 추가 액션: 우측 끝 고정]`. 제목(경로)이 길이 변동을 흡수해 도구·버튼이 제목 길이·조건부 토글에 안 밀림. 품목 추가는 품목 있을 때만(빈 잎은 EmptyState 중앙 CTA가 대신 — 배타·중복 제거).

> **검색 좌측 이전 + 목록 단일화(v0.17.0):**
> - **검색 입력 위치: 우측 헤더 → 좌측 분류 헤더 아래**(Tree `toolbar` 슬롯 신설). 전역 검색은 트리를 관통하는 행위라 트리 옆이 정합(dev 박물관과 동형). 우측 헤더는 '선택 분류 콘텐츠' 역할만.
> - **카드/목록 뷰 토글 폐기 → 목록(DataTable)만.** 잎 품목·검색 결과 모두 목록. 카드(12×6 벤토)·우측 ObjectCard 사용은 잠정 보류(가성비, 복원 백로그).
> - **검색 결과 = 목록 + `분류` 경로 컬럼**(`부자재 › 경첩`). 행 클릭=onClick(상세, 소비처 조립). 건수 `N건`은 헤더 우측 `controls`에 고정(제목존 flex:1이 흡수 → 폭 밀림 0).
> - **목록 컬럼(v0.17.1)**: 품목 · (검색 시)분류 · 핵심값 · **비고** · 상태 · 액션. `headline.note`(값-국소 배지, 예 '변경요청중')는 카드의 '값 옆 배지'를 표에선 별도 `비고` 배지 컬럼으로(`_cells` 셀 어휘 무변경, 기존 badge 타입 재사용).

### PageGrid (✅ v0.9.0 — 구현·검증 완료)

페이지 공간을 닫힌 격자로 — 위젯이 정수 칸 점유(iOS 홈 / Bento). **Bento 본질 = 가변 높이 불허:** 셀 높이는 고정(`ROW_UNIT`), 위젯은 닫힌 n×n footprint를 colSpan·rowSpan으로 *선택*. 내용이 넘치면 위젯 내부에서 처리(Tile `overflow:hidden`).

| prop | 분류 | 값 | 위치 | 노출 |
|---|---|---|---|---|
| `columns` | 값 | `2 \| 3 \| 4 \| 6 \| 12`(닫힌 열 수) | PageGrid | ✅ |
| `gap` | 값(토큰) | `'sm' \| 'md' \| 'lg'` | PageGrid | ✅ |
| `colSpan` | 값+방식 | `1..columns`(위젯 가로 칸) | `PageGrid.Tile` | ✅ |
| `rowSpan` | 값+방식 | `1 \| 2 \| 3`(위젯 세로 칸) | `PageGrid.Tile` | ✅ |
| 셀 높이(ROW_UNIT)·overflow | — | 고정(140px 잠정 — 가변 높이 불허, Tile overflow:hidden) | — | ❌ |
| `className` `style` | — | 규칙 3 | 양쪽 | ❌ |

고정: **1×1 = 절대 높이(ROW_UNIT) × 파생 폭(콘텐츠폭 ÷ columns)** — columns 선택이 곧 고정 높이에 대한 1×1 모양 선택. ROW_UNIT(현 140 잠정)은 가장 작은 자기완결 위젯(SummaryCard) 실측 높이를 8px 베이스라인에 스냅해 확정 예정(**「03」 백로그**). 1×1보다 얇은 컨트롤(검색·필터·전환탭)은 타일이 아니라 PageHeader↔그리드 사이 **컨트롤 스트립**으로(→ 「03」 11-12 화면 해부도). raw CSS grid는 Calendar 7열과 동류의 명시 예외.

### 9-1. 검증 태도: "거짓 통과"를 경계하라

> **"통과(에러 0)"는 두 가지를 뜻할 수 있다 — "검사했고 문제없음" 또는 "검사 자체를 안 함". 둘 다 0으로 보인다.**

실제 사건: 첫 `npx eslint .`가 에러 0으로 통과했는데, 규칙이 잘 작동해서가 아니라 flat config가 .ts/.tsx를 *아예 안 훑어서*였다(거짓 통과). 그냥 믿었으면 강제가 작동한다고 착각한 채 위에 다 쌓았을 것이다.

**교훈: 통과를 믿지 말고, 일부러 틀린 걸 넣어서 "막히는지"를 확인하라.**
