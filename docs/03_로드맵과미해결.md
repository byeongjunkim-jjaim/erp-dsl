# 03. 로드맵과 미해결 (어디로 가는가)

> 본 문서는 ERP No-Code 빌더 프로젝트의 **참조 문서**이다. 상위 원칙·규칙·색인은 「프로젝트 지침」을 따른다.
> 본 문서는 작업 순서·진행 상황, "올바른 방식" 문제, 미해결 지점을 다룬다. 가장 빈번히 갱신된다.

---

## 작업 순서 (로드맵)

1. ✅ 토큰 세트 확정 (theme.ts, mantine.d.ts)
2. ✅ **검증** — `npx tsc --noEmit` EXIT=0. 인터페이스 이름(`MantineThemeColorsOverride`/`MantineThemeOther`/`MantineThemeSizesOverride`) 전부 정상 확인.
3. ✅ **첫 부품 + 강제 인프라** — Button 래퍼(그물 1) + eslint 두 문(그물 2) 구현·probe 검증 완료. 별도 DSL 라이브러리 레포로 분리(특정 ERP 레포 아님).
4. ✅ **원자 분류·기준 확정** — 의미/레이아웃 원자 + 배치 프리미티브 구분, 1차 원자 집합 목록(「02. 토큰과 구현」), 판별 기준·경계 닫기 규칙·크기(밀도)·공간 장부(「01. 규칙과 구조」).
5. ✅ 의미 원자·레이아웃·프리미티브 prop 규격 확정 — 의미 원자 22 + 레이아웃 3 + 프리미티브 3, 개별 표까지(→ 「02」 원자 prop 규격 절). 입력군 3골격 + 표시·텍스트 3묶음 + 횡단 규칙 4개로 수렴(error를 입력 원자에서 제거하며 "error는 쪼갠다" 규칙이 사라져 5→4). **→ 22 원자 전부 래퍼 구현·검증 완료(tsc·eslint·강제 발화). 색·타이포·테두리는 "역할 변수 통로"(CSS 변수)로 통일(→ 「02」 역할 변수 통로 절).** (Stage5에서 Popover·Image 추가 → 의미 원자 **24**. 22는 Stage4 시점 수치.)
6. ✅ 첫 분자 — FormField 확정 — 방식 A(컨트롤을 children으로). 장식(label·withAsterisk) + 에러 표현(메시지 + 테두리)을 독점. 입력칸은 에러를 모르고, 테두리는 CSS 역할 변수(--field-border)로 FormField가 덮음(→ 「02」 분자 prop 규격). "분자가 상태를 고정한다"의 첫 사례. (description은 placeholder와 경쟁 경로라 미채택 — 「02」.) **구현·검증 완료.**
6-A. ✅ **일반 분자 확보 (kk 무관, 압도적 우선)** — 02가 적어둔 분자 후보부터: MultiSelect(Select+Chip), 로딩버튼(Button+Spinner), FileUploader, DateRangeField 등. 분자까지는 어떤 ERP에도 있는 보편 부품이므로 빈도 조사 없이 만든다. **판별 기준: "이 부품 정의에서 kk를 빼도 말이 되는가?" — 안 되면 그건 부품이 아니라 스키마다.** 
(진행: FormField·MultiSelect·DateRangeField·InputGroup·FileUploader·Pagination·IconButton·Callout ✅ — 보편 분자 **8** 구현·검증 완료. 조합 커버리지(모든 입력 원자를 FormField·InputGroup 슬롯에 꽂아 tsc 통과)도 확인. SegmentedControl·TabBar 원자 추가(의미 원자 **22** — prop 표 실측. 과거 21/23 표기는 off-by-one 오기). FileButton은 폐기(파일 업로드는 4단계 상태가 본질 → FileUploader 단일 경로). 분자 중 MultiSelect·Pagination·FileUploader는 "원자 + 격리된 Mantine 엔진(Combobox/Pagination/Progress)"으로 조립 — 도메인 무관·격리 유지, 순수 원자 조합만은 아님. **IconButton**은 Button을 아이콘 전용으로 고정한 named 분자(renderAction이 iconOnly 라우팅), **Callout**은 인라인 비휘발 안내(토스트와 의미 분리).)

6-B. ⏳ **kk 화면 빈도 지도 (병렬 가능, 유기체 착수 전 선행)** — 클로드 코드로 kk 로컬을 스캔해 "xx형 페이지 N개 / yy형 모달 N개" 식의 *현황* 지도를 받는다. 이는 *무엇이 부품인지*를 kk가 정하는 게 아니라(헌법 1번 — 도메인은 컴포넌트로 안 샘), *무엇부터 부품화할지 우선순위*를 실측으로 정하는 것이다. N≥3이 지금 추출할 후보(rule of three), N=1은 보류. 현황→부품 추출은 사람의 일(11-1).

6-C. ✅ **유기체 디벨롭 (빈도순)** — Modal·DataTable·EmptyState·PageHeader·DescriptionList·AppShell 6개 **구현·검증 완료**(「02」). 공유 패턴 2개(셀/값 type enum `text|badge|number|currency|date|actions`, actions 배열)를 공유 모듈로 선추출. AppShell은 A안(상단바 없음·로고 넷바 편입). DetailPage 통째는 템플릿(7번)으로 미룸.

6-D. ✅ **ERP 공통 셸** — 좌측 넷바(고정) + **우상단 바(고정)** + 콘텐츠. (초기 A안=상단바 없음에서 **B안=상단바 도입**으로 진화.) 프로필 카드는 넷바 하단→상단바 우측으로 이동, 알림 bell(안 읽음 점) 신설, 우측 밀착 `[알림][프로필]`. 좌측은 비움(넷바가 전체 경로를 제공해 브레드크럼 중복), 로고는 넷바 최상단(a형). 넷바 우측·상단바 하단 그림자로 둘 다 떠 있는 고정 패널. 모바일 햄버거는 상단바 좌측. kk 전용이 아니라 모든 ERP 공통(교집합).
7. ✅ **스키마 층 1차 + 1차 확장** — `FieldSpec`(name·label·type·required·placeholder·options·lookupKey·span·**pattern·mask·requiredWhen·disabledWhen**) + `FieldType` 7종(text·number·textarea·select·date·checkbox·**lookup**) + `buildZodSchema`(Zod: 타입+required+**정규식 refine+조건부 필수 superRefine**, 패턴 가드). 마스크 카탈로그(`_masks.ts`, phone). 도메인은 오직 FieldSpec[]로만 진입(헌법 1번). 함수(resolver)는 스키마 밖(코드 주입, patch 시그니처). **남은 것: min/max·교차필드(끝≥시작)·다중 술어·위저드 스키마는 데이터 계약 수령 후.**
8. ⏳ **조립 계약 + 카탈로그 문서화** — 헌법(「01. 규칙과 구조」)을 LLM 컨텍스트로 + 부품 어휘를 LLM이 읽을 지도로(「강제 vs 인지」).
9. ✅ **첫 수직 슬라이스 1종** — 고객 관리(목록 ListPage + 상세 DetailPage + 신규등록 Modal/FormSection)를 **config-driven으로 증명**(→ 「02」 템플릿/조직 절, `src/app/customers/`). 도메인 0줄 — schema(FieldSpec[])·rows·resolver만으로 작동. **남은 것: 위저드(MultiStepFormWizard)는 N단계 = 1엔진 + N스키마로 별도 증명 예정(현재는 단일 폼/목록/상세 슬라이스).**

### 실용 경고 (과도한 선제 추상화 주의)
- 모든 화면을 알기 전에 완벽히 일반적인 위젯 프레임워크를 먼저 짜면 틀린 프레임워크가 나온다.
- 같은 패턴이 **세 번째 반복될 때** 어휘로 추출한다(rule of three). 어휘가 실사용에서 *발생*하게 둔다.
- (ERP는 도메인이 닫혀 있어 이 위험이 비교적 작지만, 요구사항은 만들면서 발견된다.)
- **도메인 누수 주의 (헌법 1번의 실무 형태):** 부품은 kk를 모른다. kk(작업 중인 실제 ERP)는 이 시스템의 *부분집합*이어야 하며, kk 전용이라 다른 ERP엔 안 맞는 부품(여집합)이 생기면 안 된다. kk가 필요해서 만들고 싶어지는 부품은 부품이 아니라 *스키마*다(예: "발주항목카드"가 아니라 DataTable + 발주 스키마). 판별: 정의에서 kk를 빼도 말이 되는가.

## "올바른 방식" 문제 (가장 어려운 핵심)

자연어 설명을 받아 유한한 부품으로 위젯을 "올바르게" 조립/배치하는 판단. **이건 하나가 아니라 네 겹이고, 정의 가능한 것과 불가능한 것이 섞여 있다. 분리가 핵심.**

1. **하드 제약** — 구조적으로 허용되는 것(타입·스키마·린트가 강제). 정의 가능, 유한.
2. **기본값** — 명세가 비었을 때 적용되는 것. 사용자가 저작. 정의 가능.
3. **소프트 관습** — 권장 관용(예: "발주서엔 주소가 먼저"). 문서화하되 강제는 안 함.
4. **순수 판단** — "이 사업체엔 어떤 위젯이 맞는가". **정의 불가능. 자동화 대상 아님.**

> **촘촘하게 짜야 하는 건 1~3(정의 가능 층)이다. 4번까지 정의하려는 게 함정.**
> 4번은 정의하지 말고 사람에게 라우팅한다.

**빈칸은 LLM이 추측으로 메우게 두지 않는다.** 명세가 빈 선택 슬롯은 → 기본값을 적용하거나, 빈칸을 *드러내서 사람에게 되묻는다*. 시스템의 일은 "맞게 추측"이 아니라 "빈칸을 드러내기".

### 11-1. 규모의 역설 (현재 전략에 직결)

- **큰 규모:** 클라이언트가 *카탈로그에 맞춘다* → LLM의 판단 부담↓.
- **작은 규모(현재):** 사용자가 클라이언트와 소통하며 *사전 구조 설계 후* LLM 투입 → LLM의 판단 부담↓.
- 양 극단 모두 LLM 부담이 준다. **함의: 지금은 "올바른 방식"의 판단 엔진이 LLM이 아니라 사람(나)이다.**
- → 지금 단계엔 판단 자동화 기계를 미리 짜지 말 것. 시스템엔 *실행 충실도*만 요구한다. 정의 가능한 층(제약·기본값·관습)만 촘촘히.

### 11-2. 판단은 "정의"하지 말고 "축적"한다

순수 판단(4번)을 함수로 만들려 하지 말 것. 대신:
- 지금: 클라이언트별 "이 페이지엔 이 위젯들" 결정을 사람이 내린다.
- 축적: 그 결정들이 **이름 붙은 선례 템플릿**으로 쌓인다("가구 업체 발주 페이지는 이렇게").
- 나중(규모↑): LLM은 판단을 *계산*하는 게 아니라 *쌓인 선례 중에서 고른다*.
- → 판단은 끝내 함수가 되지 않고 **점점 커지는 예시 라이브러리**가 된다. (rule of three의 페이지 단위 적용)

### 11-3. 위젯 변형: 옵션을 쌓지 말고 위젯을 늘려라

"어떤 사업은 최고/최저, 어떤 사업은 입력칸" 같은 변동성을 한 위젯의 옵션 토글로 다 받으면, 토글 15개 = 2^15 구성 = 무한 공간의 부활(새 옷 입은). **촘촘함(정의 밀도) ≠ 유연함(옵션 개수).**
- 차라리 의견이 뚜렷한 별개 변형 몇 개로: 날씨위젯A(기본) / 날씨위젯B(최고최저 포함) / 입력형은 아예 다른 위젯.
- Button의 prop 30개 함정과 같은 원리. "유한집합" 본능이 방어선.

### 11-4. 아직 안 푼 지점

- **스키마 층 형식화 (1차 해결, 확장 진행 중).** 강점/약점 정확히: *무엇이 부품화돼야 하는지 판별하는 도메인 감각*은 강점(ERP 만들어본 자산). *그걸 형식 규칙(DSL 문법)으로 정리하는 형식화*가 약점. → 사용자가 "무엇이 부품이어야 하고 무엇이 자유여선 안 되는가"를 판단하고, Claude가 이를 스키마 문법으로 형식화한다. **1차 FieldSpec 골격 확정(→ 「02」 스키마 층). 남은 것: 정밀 검증·교차필드·조건부 필드.**
- 위저드 스키마 구조: 단계 → 섹션 → 필드 배열, 필드 타입·필수·검증규칙, 완료 추적, 임시저장, "다음" 검증 게이팅, 조건부 필드. **(현재 단일 폼/목록/상세 슬라이스까지. 위저드는 다음.)**
- ~~**검증기(Zod)**~~ → **(해결, 확장)** `buildZodSchema(FieldSpec[])` — 타입 7종+required에 더해 **정규식(`pattern`, 가드)·조건부 필수(`requiredWhen`/`disabledWhen` superRefine)**까지. 마스크(`mask`)는 입력 변환(렌더 측), pattern은 검증(스키마 측) — 별개 축. 남은 것: min/max·교차필드(끝≥시작)·다중 술어(→ 「02」 스키마 층).
- **카탈로그 진화/버전 관리** — 새 부품 추가 시 기존과의 일관성을 사람이 어떻게 보장하는가(헌법이 사람에게도 강제되는가).

소프트 관습 후보(카탈로그 문서용, 8번) — 단일선택 부품 경계: 데이터 선택(폼 제출값)은 Radio/Select, 같은 대상 뷰·모드 토글(즉시 전환, 비제출)은 SegmentedControl, 다른 구획 전환(패널 교체)은 TabBar/Tabs. 셋의 결과물이 다르므로 경쟁 경로 아님 — 단 kk처럼 단일선택에 Segmented를 남용한 사례는 의미 경계로 차단. 강제 아님(의미 판단에 사람 여지). "올바른 방식" 3번(소프트 관습) 두 번째 등록.

한 화면 primary 행동은 하나. 헤더(PageHeader)에 주 행동이 있으면 EmptyState의 action은 비운다(우리 기본 — 헤더 안정성). first-use 온보딩처럼 헤더 액션이 없으면 EmptyState action 살림. 강제 불가(부품끼리 서로 모름) 

Autocomplete — 보류 유지, 확인 대상: 측량 3건("분류 입력")이 "기존 분류 중 검색 선택"(Combobox/searchable 회귀)인지 "자유 텍스트 입력"(진짜 Autocomplete)인지 그 3건 코드 확인 후 판단.

### 11-5. 이번 세션 해결/산출

### 11-13. v0.10.0 산출 (애플식 squircle 곡률 + radius 상향)

> ERP 모서리에 애플식 통일성(squircle)을 도입. radius 스케일(sm/md/full)은 유지하되 값을 키우고, 그 위에 연속 곡률을 단일 토큰으로 얹었다. tsc·eslint EXIT=0, 크롬 화면 검증(아바타·스피너 원형 유지 + 카드 squircle 확인). 커밋·푸시·버전 태그(v0.10.0) 완료, **publish는 보류**(소비자 영향 0).

- **corner-shape(superellipse(2)=애플 squircle)을 단일 토큰으로 추가.** `theme.ts`의 `cornerShape` → `cssVariablesResolver`가 `--corner-shape` 방출(역할 변수 통로 — 색·타이포·아이콘 정렬과 동일 구조). 컴포넌트엔 prop으로 안 연다(헌법 5 — 0~100 smoothing 스칼라 노출 금지, 전역 토큰 1개).
- **박스 전용 opt-in(`squircle.css` 화이트리스트).** Button·Paper·Input·Modal·Popover·SegmentedControl에만 `corner-shape` 적용. **방향 결정:** 전역(*) 후 원형 제외(blacklist)는 빠뜨린 원형(Radio 등)이 각지므로 "박스에만 켠다(whitelist)"로 뒤집음 — 목록 밖(Avatar·Spinner·Switch·Chip·Radio·알림 점)은 자동 round. **squircle은 원/알약의 둥근 곡선까지 각지게 만들기 때문**(원→둥근 사각형). border·box-shadow·outline이 모양을 네이티브로 따라가 clip-path식 테두리·그림자 충돌 없음(MDN 확인).
- **radius 값 상향(스케일·키는 불변): sm 4→8, md 8→16.** squircle 곡률은 반경이 작으면 드러나지 않아(8px엔 곡률 바꿀 모서리가 8px뿐) md를 키워야 카드에서 보인다.
- **SummaryCard 아이콘 칩 sm 고정.** md=16이 작은 아이콘 칩(~36px)을 거의 원형으로 만들어 sm(8px)으로 복원(이전 md=8px 동치).
- **브라우저 정책:** corner-shape는 Chromium만(2026.06 ~67%, Safari/FF 미지원), **graceful fallback** — 미지원 시 평범한 둥근 모서리. radius 값은 전 브라우저 동일하므로 구조적 통일성 유지, squircle은 점진 향상 층. (사파리 혼재 환경이라 경로 A=네이티브 corner-shape 채택; 경로 B=clip-path는 테두리·그림자·no-px 충돌로 비채택.)
- **점검에서 짚은 것(동업자 판단):** ① raw-div 보조 표면(Callout·FileUploader 드롭존·DataTable 헤더)은 화이트리스트 밖이라 radius bump은 받고 squircle은 안 받음 — 경미한 cosmetic 갭, 수용. ② `Menu.tsx`의 `borderRadius: 4` 하드코딩 리터럴(토큰 아님)은 기존 드리프트, 차기 정리. ③ 소비자는 버전 bump+재설치 시 자동 반영(SSOT) — API/타입 무손상, 단 전 화면 시각 변화(둥글어짐)라 의도된 버전 업(minor)으로 표기.

## 흡수 전략 (전략 원칙 — 명문화)

> **우리는 "ERP를 닫힌 부품으로 조립한다"는 *방식*을 만들지만, 그 방식을 떠받치는 엔진·렌더링·인프라까지 전부 직접 만들지 않는다. 검증된 외부를 *흡수*한다.**

- **이미 굴지의 기업들이 같은 "방식"으로 접근했다** — JSON/스키마로 UI를 구동하는 스키마-드리븐 렌더링(예: Vercel **json-render** 류), 닫힌 컴포넌트 레지스트리, 디자인 토큰 시스템. 이건 우리만의 발명이 아니라 *업계가 수렴 중인 패턴*이다.
- **그래서 "내가 전부 만든다"는 욕심을 버린다.** 엔진(스키마→렌더 변환·resolver·런타임)은 검증된 외부로 **흡수**하고, 우리는 *대체 불가한 고유 가치*에만 집중한다:
  - **디자인 토큰 + 닫힌 부품 카탈로그**(이 DSL의 헌법·강제 3층) — 통일성의 원천.
  - **`FieldSpec`/스키마 층**(도메인 진입로) — ERP 도메인 형식화.
  - **ERP 특화 부품**(HierarchyExplorer·ObjectCard·정산/발주 조립 패턴) — 도메인 자산.
- **판별 기준:** "이게 우리 ERP-DSL의 고유 가치인가, 아니면 누구나 쓰는 범용 엔진인가?" 후자면 흡수(외부 채택), 전자면 직접. (헌법 4의 "큐레이션"이 *부품* 차원이라면, 흡수 전략은 *인프라* 차원의 같은 사상 — 우리가 통제할 경계와 빌려올 경계를 가른다.)
- **현 단계:** kk 건은 지금 구조(자체 렌더)로 마무리하고, 그 이후 json-render 류로 엔진을 흡수하는 전환을 디벨롭한다. 토큰·FieldSpec·ERP 부품은 흡수 후에도 우리 것으로 남는다(엔진만 교체, 어휘·문법·도메인은 유지).

### 11-16. FieldGrid(장표) + 폰트 스케일(접근성) + 줌 검수 (커밋, **publish 보류**)

> kk 실수요에서 **"엑셀 인쇄물 그대로의 테두리 격자 문서(도면)"**가 나왔다. 판단 결론(주객전도 경계): *임의 픽셀 충실 재현*은 토큰-닫힘 DSL과 정반대라 DSL에 안 넣지만, **"테두리 친 셀 격자 안에 라벨/입력칸을 배치하는 장표(帳票)"는 한국 ERP·관공서 양식의 보편 포맷**(kk 빼도 성립)이라 범용 부품으로 확보. + 고령 클라이언트 대비 폰트 스케일 레버. tsc·eslint·build EXIT=0, dev 박물관 시각 검증.

**신규 유기체 (FieldGrid):**
- 테두리 2D 셀 격자. 셀 = 고정 라벨 / 스키마 필드(`FieldSpec` 재사용) / 이미지 / 빈 칸, `colSpan`·`rowSpan` 병합. 머리표(라벨:값)·명세표(헤더+값 행)·대분류 밴드(라벨 colSpan=전체) 다 같은 `rows: FieldGridCell[][]` 모델. **라벨↔입력 쌍 강제 아님**(셀 종류 독립).
- **mode `edit`/`read` — 셀 박스 기하 모드 불변, 값 렌더러만 스왑.** 확인=작성과 *같은 입력칸을 그대로* 쓰되 편집만 차단(`inert`) → "차이는 편집 가능 여부 하나뿐"(폰트·높이·인셋·값 표현 구조적 동일). 별도 텍스트 렌더(`renderCell`) 제거. 입력칸은 자체 테두리·radius·여백을 벗겨 **셀 테두리에 흡수**(작성/확인 같은 크기), 모든 셀 라인박스를 `--typo-body-*`로 못박아 행 높이 균일.
- 에러 = `--field-border` danger + Tooltip(기하 보존, FormField 안 씀 — 라벨이 별도 셀). `mask` 파리티. **명시 예외**: raw CSS grid(`fieldgrid.css`, 임의 N열+rowSpan — Calendar 7열·Modal raw flex 동류). 모서리 radius 0(엑셀 평면). prop 표는 `_catalog` 위임.

**폰트 스케일(접근성 — 기존/크게/아주크게):**
- 타이포·간격이 *둘 다 rem*, line-height 단위 없는 배수 → **루트 `font-size` 하나로 글자+간격+Mantine 컨트롤이 비율 그대로 함께 줌**(상대 비율 고정). 6단계 타이포·컴포넌트 0줄 수정. 포털(Modal/Popover/Toast)도 `:root` 상속. `--mantine-scale`은 rem-기명 spacing 스케일 불확실로 배제, 루트 font-size 채택.
- `fontscale.css`: `:root[data-font-scale="large"]→112.5%(18px)` / `"xlarge"→125%(20px)`. 소비 앱은 `<html data-font-scale="large">` 1회(플래시 0). dev 좌측 토글로 검증.
- **전역 줌 검수**: 안 따라오는 리터럴 px 전수 점검. **CountBadge가 유일 실문제**(18/11/8px → 옆 라벨만 커지고 빨강 N만 쪼그라듦 = 접근성 역행) → **rem화**(1.125/0.6875/0.5rem). Tree 인라인 20px·터치타깃 44px는 고정이 맞음(ROW_H 36 내부정합 / 절대 하한). 모바일 하단탭 56px·Image 썸네일 고정폭은 선택적 후속(미관·모바일 한정).

**Tree 라벨 말줄임:** Tree 표시 라벨이 길면 줄바꿈 → 고정 행(ROW_H 36) 넘어 다음 행과 겹침. → **한 줄 `nowrap`+`…`+`title` 호버**(VS Code·Breadcrumb 선례). `Text` 원자가 truncate 미노출 → 격리 구역에서 `--typo-body-*` role 변수로 직접.
